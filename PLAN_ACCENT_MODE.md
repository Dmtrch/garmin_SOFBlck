## Идея
На главном экране одиночное нажатие **DOWN/UP** активирует режим «акцент» и
последовательно выделяет один из 12 показателей по периметру: его значение
рисуется шрифтом ×2, **сдвигается ближе к центру** и выводится **на чёрной
подложке по размеру значения** (перекрывает фон и соседние метки), и обновляется
со своей частотой. Переход: **DOWN** — по часовой стрелке, **UP** — против.
Предыдущий выделенный показатель возвращается к базовому шрифту, позиции и частоте
обновления. **BACK** выходит из режима (всё к базовому виду).

## Базовое обновление (находка)
Главный экран **уже** перерисовывается раз в 1 с: `TactixView.onShow()` запускает
`mTimer` на 1000 мс (`onTick → requestUpdate`), `onHide()` его гасит. Значит:
- Базовая частота перерисовки = **1 с** для всех показателей (значения
  перечитываются в каждом `onUpdate`). Отдельный таймер для режима акцента **не
  нужен** — переиспользуем существующий 1-секундный `mTimer`.
- «Частота в режиме акцента» (таблица ниже) реализуется как **период
  перечитывания значения** выбранного показателя из источника (кэш по таймстампу).
  Перерисовка кадра остаётся 1 с; крупное значение между опросами берётся из кэша.
- **При деактивации** (BACK / уход с показателя / выход) частотный кэш
  сбрасывается, и все показатели снова обновляются базово каждые 1 с.

## Три уровня частоты (важно не смешивать)
1. **Перерисовка экрана** (`onUpdate`) = 1 с через `mTimer`. Кадр, стрелки,
   секунды, все метки. **Фиксировано 1 с всегда** — режим акцента это не меняет.
2. **Чтение значения нашим кодом** — вызовы `*History()/getInfo()/getSystemStats()`
   в `onUpdate`. Это и есть «частота» из таблицы (через кэш по таймстампу для
   акцентированного показателя). Базово — каждый кадр (1 с).
3. **Физический опрос датчиков** — **мы их постоянно НЕ запрашиваем.** Методы
   `ActivityMonitor`/`SensorHistory`/`getSystemStats` читают данные, которые
   система уже накопила в фоне (история HR/высоты/давления пишется системой с
   периодом ~минуты; шаги/калории считаются фоном; заряд меняется медленно). Это
   дешёвое чтение из памяти, датчик не включается и энергию почти не тратит.

**Единственное исключение** — реальный live-опрос датчика через
`Sensor.enableSensorEvents` + `Sensor.setEnabledSensors`. Это реально включает
датчик (энергия!). Нужно только для честного ежесекундного пульса (idx 1) — см.
шаг 4. Для остальных показателей уровень 3 не задействуется вообще.

**Следствие для таблицы:** большие периоды (300/30 с) у idx 0,3–11 ничего не
«теряют» — системный кэш этих значений и так обновляется реже; мы лишь реже
перечитываем его. Параметр «1 с» у пульса станет настоящим только при включённом
Sensor HR; иначе это частота перечитывания того же исторического значения.

## Разрешённые развилки (согласовано)
1. **Кнопки:** двойное UP→будильники и двойное DOWN→таймеры сохраняются.
   Одиночное UP/DOWN определяется по таймауту `DOUBLE_PRESS_MS` (~500 мс) и
   переключает акцент. Цена — задержка ~0.5 с на переключение (как у BACK/SELECT
   сейчас).
2. **Выход:** одиночный **BACK**, когда режим активен, сбрасывает акцент
   (`accentIndex = -1`) вместо ухода на эко-экран. Когда режим выключен — текущее
   поведение BACK без изменений. Двойной BACK (навигация) не трогаем.
3. **Шрифт ×2:** через `Graphics.AffineTransform` в `placeText()` (буфер уже
   создаётся и поворачивается — добавляем `scale(2.0, 2.0)`).

| 1   | 45°  | Пульс                   | **Sensor HR (live)**                  | **1 с**      |

| 4   | 132° | Температура             | `Weather.getCurrentConditions`        | 300 с        |

> (`requestUpdate`). Фактическая свежесть данных из `SensorHistory`/
> `ActivityMonitor` ограничена самим источником (барометр/альтиметр пишут историю

> настоящего ежесекундного пульса нужен `Sensor.enableSensorEvents` с
> `Sensor.SENSOR_HEARTRATE` — см. шаг 4 (опционально).

### 3. `source/TactixView.mc` — отрисовка ×2 + частотный кэш
- `drawDataLabels()` читает `app.accentIndex`. Каждый из 12 вызовов `placeText`
  получает признак «акцент» (idx совпал) → передаёт `scale = 2.0`, иначе `1.0`.
  Чисто: завести локальный счётчик idx или явный индекс на каждый вызов.
- **Частотный кэш значения акцентированного показателя:** для idx == accentIndex
  строка значения перечитывается из источника только если
  `System.getTimer() - app.accentValueTs >= ACCENT_PERIODS_MS[idx]`; иначе берётся
  `app.accentValueStr`. При опросе — обновить `accentValueStr`/`accentValueTs`.
  Базовый кадр продолжает тикать 1 с — это лишь ограничивает частоту опроса
  источника, не перерисовку. Остальные 11 показателей читаются как сейчас.
- **Акцентированное значение (×2) рисуется по-особому** — сдвиг к центру + чёрная
  подложка, чтобы не вылезать за край и перекрывать соседей:
  - `placeText(...)` получает параметры `scale as Float` и `opaque as Boolean`
    (или отдельный метод `placeAccentText`).
  - **Шрифт ×2:** размер буфера `bmpW/bmpH` × `scale`; в `AffineTransform`
    добавить `t.scale(scale, scale)` перед поворотом; центр буфера и точка вывода
    пересчитываются с учётом увеличенного размера.
  - **Чёрная подложка по размеру значения:** буфер заполняется чёрным
    (`bmpDc.setColor(COLOR_BLACK, COLOR_BLACK); bmpDc.clear()`), текст — белым
    поверх. Размер подложки = размер текста + паддинг (т.е. «по размеру выводимых
    значений»). На экран bitmap выводится непрозрачным прямоугольником, перекрывая
    фон и соседние метки. (У обычных меток фон остаётся `COLOR_TRANSPARENT`.)
  - **Сдвиг к центру:** для акцентированного использовать уменьшенный радиус
    `R_accent` (< базового `R = s(119)`) на той же угловой позиции — показатель
    остаётся «на своём месте» по кругу, но ближе к центру, освобождая периметр.
    Конкретное значение подобрать так, чтобы ×2-bbox не выходил за экран на 260 px
    (напр. ориентир `R_accent ≈ s(85)`; финально — по факту на симуляторе).
  - **Поворот:** ориентация **по кругу** (как у обычных меток) — `rotAngle`
    сохраняется, акцентированное значение поворачивается так же, как обычные
    метки на этой угловой позиции. (Решено.)
- **Порядок отрисовки:** акцентированный показатель рисовать **последним**, после
  цикла обычных 12 меток (отдельным проходом), чтобы его чёрная подложка легла
  поверх соседей, а не была перекрыта ими.

## Шаг 4 — live-датчики (пульс + барометр), единый Sensor-диспетчер
Три показателя берутся live: idx 1 (HR), idx 2 (altitude), idx 8 (pressure).
Все используют `Toybox.Sensor`, который в проекте уже занят магнитометром
(heading для компаса/пеленга, `onCompassSensor`). Нельзя держать два независимых
`enableSensorEvents` — он один на приложение. → **рефактор в единый диспетчер.**

**Состояние в `TactixApp`:**
- `var currentHr as Number? = null;`
- `var currentAltitude as Float? = null;`
- `var currentPressure as Float? = null;`

**Метод `_syncSensors()`** — управляет подпиской и набором сенсоров:
- `heading`, `altitude`, `pressure` — **positional/onboard, приходят в `Info`
  автоматически** при `enableSensorEvents`, без `setEnabledSensors` (подтверждено:
  текущий компас читает `info.heading` без `setEnabledSensors`). Значит для высоты
  (idx 2) и давления (idx 8) **дополнительного включения не требуется** — нужна
  лишь активная подписка.
- `heartRate` — **требует явного включения** ANT/onboard HR-сенсора. Использовать
  **точечные** `Sensor.enableSensorType(Sensor.SENSOR_HEARTRATE)` /
  `Sensor.disableSensorType(Sensor.SENSOR_HEARTRATE)` (API 3.2.0 — ок для API
  6.0.0), **а НЕ** `setEnabledSensors([...])`: последний перезаписывает весь набор
  и может сбить heading. `enableSensorType/disableSensorType` не трогают другие
  типы (подтверждено докой).
- Подписка: если нужен хоть один источник (компас/пеленг/любой акцент-датчик) →
  `Sensor.enableSensorEvents(method(:onSensor))`; если ничего —
  `enableSensorEvents(null)` (как сейчас в `_releaseMagnetometer`).
- При входе на idx 1 → `enableSensorType(HEARTRATE)`; при уходе с idx 1 →
  `disableSensorType(HEARTRATE)`.

**Единый колбэк `onSensor(info as Sensor.Info)`** заменяет `onCompassSensor`
(события приходят с частотой **1 Гц** — подтверждено докой):
- если `(compassActive || bearingActive)` и `info.heading != null` → текущая
  логика компаса (`compassHeading`, `compassHeadingMs`, `compassError`);
- если `accentIndex == 1` и `info.heartRate != null` → `currentHr`
  (`info.heartRate` — `Number?`);
- если `accentIndex == 2` и `info.altitude != null` → `currentAltitude`
  (`info.altitude` — `Float?`, метры над уровнем моря);
- если `accentIndex == 8` и `info.pressure != null` → `currentPressure`
  (`info.pressure` — `Float?`, **Паскали**; для отображения `/100.0` → гПа/мбар,
  как сейчас делает код с history);
- `WatchUi.requestUpdate()`.

**Точки вызова `_syncSensors()`:** заменяет/дополняет `_ensureMagnetometer()` и
`_releaseMagnetometer()`; вызывается из `accentNext/Prev/accentOff`,
`startBearing/stopBearing`, переключения компаса, `resumeSensors/suspendSensors`.

**Возврат при переходе:** при переключении на другой показатель датчик
предыдущего показателя гаснет. `_syncSensors()` пересобирает набор по *новому*
`accentIndex`, поэтому пульсометр (уход с idx 1) или барометр (уход с idx 2/8)
выпадает из набора и отключается — если его не держит компас/пеленг. Значение
бывшего показателя снова читается из `*History()`, шрифт возвращается к базовому,
а частотный кэш (одноэлементный) обнуляется. То же при `accentOff()`.

**Отрисовка (`TactixView.drawDataLabels`)** для idx 1/2/8:
- если показатель **выбран** (accentIndex совпал) → live из `currentHr` /
  `currentAltitude` / `currentPressure`. Для idx 2/8 при `null` (нет GPS/позиции)
  → **fallback** на `getElevationHistory()`/`getPressureHistory()`. Частотный кэш
  (5 с для барометра) троттлит частоту фиксации значения в отображение, т.к.
  колбэк приходит 1 Гц;
- если **не выбран** → как сейчас, из `*History()` (датчик через Sensor API не
  держим, экономим).

> Энергозатраты: пульсометр дорогой — включаем только пока выбран idx 1.
> Барометр маломощный (обычно уже активен) — включаем на idx 2/8 дёшево.

### Сверка с ConnectIQ SDK 9.1.0 (факты из doc/)
Источник: `doc/Toybox/Sensor/Info.html`, `doc/docs/Core_Topics/Sensors.html`.
- **Поля `Sensor.Info`** (все nullable):
  - `heartRate as Number?` — уд/мин;
  - `altitude as Float?` — высота над уровнем моря, м («Positional Altitude»);
  - `pressure as Float?` — барометрическое давление, **Паскали**;
  - `heading as Float?` — азимут на истинный север, радианы (уже используется);
  - также есть `temperature/speed/cadence/power/accel/mag/oxygenSaturation`.
- **Частота событий** `enableSensorEvents` = **1 Гц** (1 событие/с). Поэтому 5-с
  период для барометра реализуем как throttle отображения, не как частоту колбэка.
- **Включение:** `heartRate` требует `setEnabledSensors`/`enableSensorType`;
  `altitude`/`pressure`/`heading` — positional/onboard, идут в `Info` без явного
  включения. Точечно управлять HR: `enableSensorType`/`disableSensorType`
  (API 3.2.0), а не перезапись `setEnabledSensors`.

### ⚠️ Важно про давление (idx 8) — требует решения
`Sensor.Info.pressure` — это давление, **калиброванное к уровню моря**
(sea-level): дока прямо пишет, что сначала нужна **GPS-высота**, затем ambient
давление пересчитывается в калиброванное. Следствия:
1. Без активного GPS-фикса live `pressure` может быть `null`/неточным. В проекте
   GPS включается отдельной кнопкой (`toggleGps`) и обычно выключен → live
   давление может не работать на idx 8.
2. Текущий код берёт давление из `SensorHistory.getPressureHistory()` — это
   **ambient (raw)** давление, доступно без GPS. Значения live (sea-level) и
   history (ambient) **различаются** — на idx 8 число «прыгнет».

**Решение (согласовано): Live + fallback History** для обоих:
- **idx 8 (давление):** читать `Sensor.Info.pressure` (live, sea-level, Па); если
  `null` → fallback на `SensorHistory.getPressureHistory()` (ambient). Отображение
  `/100.0` → гПа.
- **idx 2 (высота):** читать `Sensor.Info.altitude` (live); если `null` → fallback
  на `SensorHistory.getElevationHistory()`.
- **Нюанс:** при переходе live↔fallback значение может скакнуть (sea-level vs
  ambient для давления; разные источники высоты) — это принято. Можно по желанию
  пометить fallback-значение визуально, но в базовом плане — без пометки.
- Live-значение пишется в `currentPressure`/`currentAltitude` колбэком `onSensor`;
  в `drawDataLabels` для idx 2/8: `currentX != null ? currentX : <history>`.

## Риски / что может сломаться
1. **×2 вылезает за край / перекрывает соседей.** Решается дизайном: акцентированное
   значение сдвигается к центру (`R_accent`) и рисуется на **чёрной подложке** по
   размеру текста, перекрывая соседние метки; отрисовывается последним. Остаётся
   подобрать `R_accent` и паддинг так, чтобы ×2-bbox не выходил за экран на 260 px.
   → Проверить на `fenix8solar47mm` (260) и `fenix847mm` (454).
2. **Задержка 0.5 с** на каждое переключение (плата за сохранение двойных
   жестов). Поведение ожидаемое, но ощущается — проверить UX.
3. **BACK-переопределение.** Легко сломать выход на эко-экран/навигацию. Тест:
   режим выкл → BACK работает как раньше; режим вкл → одиночный BACK гасит акцент,
   двойной BACK всё ещё открывает навигацию.
4. **Расход батареи** от таймера 1 с (пульс/высота). Таймер активен только в
   режиме акцента — приемлемо.
5. **Регрессия компаса/пеленга.** Рефактор `onCompassSensor → onSensor` и
   `_ensureMagnetometer → _syncSensors` затрагивает рабочую логику компаса. Тест:
   компас и пеленг работают как раньше при включённом/выключенном акценте.
6. **Буфер ×2** — `createBufferedBitmap` может вернуть null на большом размере →
   обработать (уже есть guard на null).
7. **Sensor API (сверено с SDK 9.1.0):** один `enableSensorEvents` на приложение
   → единый `onSensor`. Поля подтверждены: `heartRate:Number?`, `altitude:Float?`,
   `pressure:Float?`(Па), `heading:Float?`(рад). HR включать точечно через
   `enableSensorType/disableSensorType` (НЕ `setEnabledSensors`, чтобы не сбить
   heading). Колбэк 1 Гц. Остаточный риск — проверить на устройстве, что включение
   HR не глушит heading во время активного компаса.
8. **Live-давление зависит от GPS** (sea-level калибровка) и отличается от
   текущего ambient из `getPressureHistory()` — см. раздел «⚠️ Важно про давление».
   Аналогично `altitude` («Positional») может быть `null` без позиции → нужен
   fallback на `getElevationHistory()`.

## Тесты (ручные, в симуляторе)
- DOWN ×1 из выкл → акцент на «заряд дней» (idx0), шрифт ×2.
- DOWN ×1 ещё → idx1 «пульс» ×2, idx0 вернулся к базовому; пульс обновляется ~1 с.
- UP ×1 → шаг против часовой, частота/шрифт предыдущего сброшены.
- Полный круг DOWN ×12 → возврат к idx0 (цикл).
- BACK ×1 в режиме → всё к базовому, режим выкл.
- Двойное DOWN → таймеры; двойное UP → будильники (не сломаны).
- BACK ×1 вне режима → эко-экран; BACK ×2 → навигация.
- Сборка на 260 и 454 px — проверить, не вылезает ли ×2 за край.
- **Live-датчики:** на idx 1 — пульс обновляется ~1 с (Sensor HR); при уходе с
  idx 1 пульсометр гаснет. На idx 2/8 — высота/давление свежие (~5 с) из Sensor.
- **Компас + акцент одновременно:** включить компас/пеленг, затем выбрать idx 1
  (или 2/8) — heading и live-значение приходят оба; при `accentOff` heading
  остаётся, HR/барометр гаснут (если не нужны компасу).
